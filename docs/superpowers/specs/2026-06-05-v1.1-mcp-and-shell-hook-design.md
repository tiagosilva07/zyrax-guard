# Invoke Guard v1.1 — MCP server + shell-hook (Design)

**Date:** 2026-06-05
**Status:** Design — awaiting review before implementation plan
**Builds on:** v1 (released v0.1.0). Both features are new **surfaces** over the existing
`internal/check.Orchestrator.Check(ctx, name, version) verdict.Result` — **no engine,
check, or verdict changes.**

---

## 1. Goal

Make Guard's protection **proactive and transparent** for AI coding agents and humans:
1. **MCP server** (`invoke-guard mcp`) — lets an MCP-capable agent (Claude Code, Cursor,
   Windsurf) call a `check_package` tool to vet a dependency *before* it runs an install
   command.
2. **Shell-hook** (`invoke-guard init <shell>`) — transparently routes `npm install` through
   Guard for any terminal user or agent, without changing how they invoke npm.

Both stay **stdlib-only** (zero third-party dependencies — on-brand for a supply-chain tool).

## 2. Non-goals (v1.1)

- No MCP SDK dependency (implement the minimal protocol over stdio directly).
- No batch/scan MCP tool — just `check_package` (YAGNI).
- No yarn/pnpm/pip/cargo interception — `npm` only (matches the v1 ecosystem; broader
  wrappers arrive with v1.2 ecosystems).
- No engine/check/verdict/reporter changes.

## 3. Architecture

```
                       cmd/invoke-guard (subcommand dispatch)
                          │            │
                       mcp │            │ init <shell>
                          ▼            ▼
                  internal/mcp     internal/hook
                  (stdio JSON-RPC)  (shell snippet templates)
                          │
                          ▼  Checker interface
                  internal/check.Orchestrator.Check()   ← unchanged v1 core
```

A tiny seam keeps the MCP server testable without network:

```go
// internal/mcp
type Checker interface {
    Check(ctx context.Context, name, version string) verdict.Result
}
```

`*check.Orchestrator` already satisfies `Checker` (it has that exact method). The `mcp`
subcommand wires the real orchestrator; tests inject a fake.

## 4. MCP server — `invoke-guard mcp`

### Transport & protocol
- **stdio**, **newline-delimited JSON-RPC 2.0** (MCP's stdio framing: one JSON object per
  line on stdin/stdout; logs go to stderr only).
- Implements the minimal MCP method set:
  - `initialize` → returns `protocolVersion`, `serverInfo` (`{name:"invoke-guard", version}`),
    and `capabilities: { tools: {} }`.
  - `notifications/initialized` → accepted, no-op.
  - `tools/list` → returns the single `check_package` tool with its JSON input schema.
  - `tools/call` (name `check_package`) → runs the check, returns the result.
  - Unknown method → JSON-RPC error `-32601` (method not found). Malformed JSON → `-32700`.

### The `check_package` tool
Input schema (JSON Schema):
```json
{
  "type": "object",
  "properties": {
    "name":      { "type": "string", "description": "package name to check" },
    "version":   { "type": "string", "description": "optional; defaults to the latest" },
    "ecosystem": { "type": "string", "enum": ["npm"], "default": "npm" }
  },
  "required": ["name"]
}
```
Handler:
1. Validate `name` is present and non-empty (reject otherwise with a tool error result).
2. `res := checker.Check(ctx, name, version)`.
3. Return a `tools/call` result whose `content` is one `text` block containing a
   plain-language summary the agent can act on, e.g.:
   `BLOCK invoke-guard: "reqeust" — looks like a typo of "request" … Do not install; did you mean "request"?`
   followed by the structured JSON (`{verdict,name,version,score,signals,suggestion}`) so the
   agent can branch programmatically.
4. **A BLOCK/WARN is NOT a JSON-RPC error and NOT `isError:true`** — it's a valid check
   result the agent must heed. `isError` is reserved for genuine tool failures (e.g. the
   registry was unreachable AND no verdict could be formed). The text makes the
   recommendation explicit ("Do not install" on BLOCK).

### Agent registration (documented in README)
```
claude mcp add invoke-guard -- invoke-guard mcp
```
(or the equivalent `mcpServers` entry for Cursor/Windsurf). The agent then calls
`check_package` before running an install.

### Files
- `internal/mcp/server.go` — the stdio read/dispatch/write loop + JSON-RPC types.
- `internal/mcp/checkpackage.go` — the `check_package` tool definition + handler.
- `internal/mcp/server_test.go` — scripted request/response tests with a fake `Checker`.

## 5. Shell-hook — `invoke-guard init <shell>`

`invoke-guard init <bash|zsh|powershell>` prints a snippet the user adds to their shell rc:
```bash
# ~/.zshrc
eval "$(invoke-guard init zsh)"
```
The snippet defines an `npm` wrapper:
- If invoked as `npm install|i|add <args…>` → run `invoke-guard install <args…>` (which
  checks then runs the real install on pass; blocks otherwise).
- Otherwise → pass through to the **real** npm via `command npm "$@"` (bash/zsh) /
  `(Get-Command -CommandType Application npm)` (PowerShell) — **never recurses**.
- Unknown shell arg → error to stderr + non-zero exit.

### Correctness risks (explicitly handled + tested)
- **Recursion:** the wrapper must call the real binary (`command npm`), never `npm`.
- **Non-install commands must pass through untouched** (`npm run build`, `npm ci`, `npm test`
  …) — only `install`/`i`/`add` are intercepted.
- **Arg forwarding:** all args/flags forwarded verbatim to the real npm on pass-through.

### Files
- `internal/hook/hook.go` — `Snippet(shell string) (string, error)` returning the per-shell
  template (bash/zsh share one POSIX template; powershell its own).
- `internal/hook/hook_test.go` — render each shell, assert structure (defines wrapper, routes
  install/i/add, uses `command npm`/real-npm lookup, passes others through); one bash
  integration test that sources the snippet and verifies the function + routing.

## 6. CLI changes

`cmd/invoke-guard/main.go` gains two subcommands in the dispatch switch:
- `mcp` → `mcp.Serve(os.Stdin, os.Stdout, realOrchestrator)` (blocks serving until stdin EOF).
- `init <shell>` → print `hook.Snippet(shell)` to stdout (exit 2 on a missing/unknown shell).

Usage text updated to list both. `--help`/`--version` unchanged.

## 7. Testing

- **MCP:** feed a scripted `initialize` → `tools/list` → `tools/call` byte stream into
  `mcp.Serve` with a fake `Checker`; assert each JSON-RPC response (matching `id`, the tool
  schema in `tools/list`, and that a BLOCK verdict renders a clear "do not install" text
  result with `isError` absent/false). Malformed-line → `-32700`; unknown method → `-32601`.
- **Hook:** golden/structural assertions on each rendered snippet; a bash integration test
  (`bash -c 'eval "$(... init bash)"; type npm'`) confirming the wrapper is defined and that
  `install` routes while `run` passes through (using a stub `invoke-guard`/`npm` on PATH).
- Full suite stays green; `invoke-guard` stays CGO-free and dependency-free
  (`go.mod` require block remains empty).

## 8. Resolved decisions

1. **stdlib MCP over stdio**, newline-delimited JSON-RPC 2.0 — no SDK, no deps.
2. **Single `check_package` tool** (no batch/scan in v1.1).
3. **Shells: bash, zsh, powershell** (fish is a fast-follow).
4. **npm-only interception** (`install`/`i`/`add`); everything else passes through.
5. **BLOCK/WARN are normal tool results**, not MCP errors — the agent decides, with an
   explicit textual recommendation.
6. **Checker seam** keeps the MCP server testable without network; `*Orchestrator` satisfies it.

## 9. Open questions (resolve during implementation)

1. **MCP `protocolVersion` value** — emit the current MCP spec date string the major clients
   accept; confirm against Claude Code/Cursor at implementation time (echo the client's
   requested version if provided, else a known-good default).
2. **PowerShell real-npm resolution** — confirm `Get-Command -CommandType Application npm`
   reliably finds the real `npm.cmd`/`npm` and that the wrapper function name doesn't shadow
   it; adjust if needed during implementation.
3. **`tools/call` structured output** — whether to also populate MCP `structuredContent`
   (newer spec field) in addition to JSON-in-text; default to JSON-in-text for broad client
   compatibility, add `structuredContent` if trivial.
